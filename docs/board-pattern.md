# Optional: plan your content on a GitHub Project board

The radar generates idea cards. The polish skill turns them into posts. But between "I have an idea" and "I clicked publish" there's usually a queue вҖ” drafted, scheduled, parked, killed. A GitHub Project board is a clean place to track that queue without paying for Notion or Trello.

This doc describes the pattern. You build the board yourself. It takes about 5 minutes.

## Why a board (vs. just `content-plan.yaml`)

- The YAML is your **editorial frame** (scope, tracks, what you cover) вҖ” slow-moving.
- The board is your **content pipeline** (this specific idea, this week, this status) вҖ” fast-moving.
- Keep them separate. Editing YAML every time a post moves columns gets old fast.

Five is enough. More is theater.

- **Idea** вҖ” generated by the radar's weekly ideas run, or added manually.

- **Scheduled** вҖ” pushed to LinkedIn drafts (or your scheduling tool). Has a target publish date.
- **Posted** вҖ” live. Includes a link back to the public post.
- **Parked** вҖ” good idea, wrong week. Lives here forever until you delete it or pull it back.

## Custom fields

Four fields, all single-select:

| Field | Options |
|-------|---------|
| **Priority** | 1, 2, 3 |
| **Track** | A, B, C, D вҖ” match these to your `content-plan.yaml` tracks |
| **Format** | case-study, opinion, how-to, announcement, observation |
| **Target date** | Date field |

Add more if you want (audience tag, channel, source URL). Resist the urge.

## Create the board

UI is faster than the API for one-off setup.

1. Go to https://github.com/users/YOUR-USERNAME/projects вҶ’ **New project** вҶ’ **Board** layout.
2. Name it "Content pipeline" or similar.
3. Default Status field already has To-do / In Progress / Done. Rename the options to **Idea / Drafted / Scheduled / Posted / Parked** (Settings вҶ’ Status вҶ’ edit options).
4. Add the three custom single-select fields above (Priority, Track, Format). Click **+** in the field list.
5. Add the Date field for Target date.
6. Done.

## Linking it to signal-engine

Two natural patterns. Pick one. Or both.

**Pattern 1: ideas вҶ’ issues вҶ’ board**

Each Monday the radar generates idea cards in `digests/YYYY-WW.md`. Manually copy the cards you want to act on into GitHub Issues in your signal-engine repo (or wherever). Add each issue to the project board. Set Track + Format + Priority.

Trade-off: manual. But it's a 30-second copy-paste per card, and you're filtering anyway.

**Pattern 2: file-per-post in `_workdir/posts/`**

When the polish skill produces a post, the file lands at `_workdir/posts/<slug>.md`. Open an issue titled the same as the slug, link the file in the issue body, move the card to **Drafted**. After `linkedin_post.py` runs, move to **Scheduled** and paste the LinkedIn URN into the issue.

Trade-off: more housekeeping, tighter trace from idea to live post.

## Automating the sync (later)

If you want the board to auto-update when the radar runs, you can write a small script that reads the project state via `gh project item-list <number> --owner <user> --format json` and your `content-plan.yaml` and reconciles. The official author runs a private version of this. It's not hard but it's about an evening of work and very specific to your setup, which is why it isn't shipped in this repo.

If you build one, PR it вҖ” generic enough to drop in here is a real contribution.

## Anti-pattern: don't track posts in YAML

You will be tempted to add a `status: drafted` field to each item in `content-plan.yaml`. Don't. YAML is for your slow editorial frame. The board is for the weekly pipeline. Keep them apart and the engine stays simple.
